# Phase 2: GraphStore 增强 — 探索数据导入与图谱构建

## 目标

将 Phase 1 OfflineExplorer 产出的结构化 JSON 导入 Neo4j GraphStore，构建带有页面类型分类的 ShoppingGraph，支持语义合并和增强的轨迹查询。

## 现状分析

### 已有基础设施

| 组件 | 文件 | 行数 | 关键能力 |
|------|------|------|---------|
| GraphStore | `memory/graph_store.py` | 433 | Neo4j CRUD, TaskIndex, N-gram fallback |
| MemoryManager | `memory/memory_manager.py` | 1146 | 三层匹配, 轨迹上下文注入 |
| StateManager | `memory/state_manager.py` | 101 | state_id 计算, prev/current 跟踪 |
| MemoryStore | `memory/memory_store.py` | 666 | FAISS 2048d, embedding-3 |
| TaskIndex | `memory/task_index.py` | 108 | FAISS 1536d, 任务语义搜索 |
| OfflineExplorer | `memory/offline_explorer.py` | ~400 | VLM 自主探索, 页面分类, JSON 导出 |

### 当前 GraphStore 的局限

1. **UIState 节点无 page_type 属性** — 无法按页面类型查询和过滤
2. **无批量导入接口** — 只有 `add_state_transition()` 逐条插入，无批量导入
3. **轨迹查询硬编码** — `get_task_trajectory()` 用 15 个硬编码 MATCH 子句，不支持变长路径
4. **无页面去重/合并** — 相同类型的相似页面会重复创建 UIState 节点
5. **TaskTarget 缺少探索元数据** — 不区分在线执行轨迹 vs 离线探索轨迹

### 探索数据格式（Phase 1 产出）

```json
// {app}_pages_{ts}.json
{
  "app": "淘宝",
  "pages": [{
    "page_type": "home",
    "summary": "淘宝首页 (Tab: 首页,分类,购物车,我的)",
    "elements": {"search_box": {...}, "tab_buttons": [...]},
    "screenshot_hash": "abc123",
    "app": "com.taobao.taobao"
  }]
}

// {app}_trajectory_{ts}.json
{
  "app": "淘宝",
  "steps": [{
    "step": 1,
    "page_type": "home",
    "page_summary": "淘宝首页",
    "action": {"_metadata": "do", "action": "Tap", "element": [500, 100]},
    "thinking": "...",
    "timestamp": "2026-05-03T..."
  }]
}
```

## 任务拆解

### Task 2.1: UIState Schema 增强

**目标**: UIState 节点新增 `page_type` 和 `semantic_summary` 属性。

**文件**: `phone_agent/memory/graph_store.py`

**变更**:
1. 在 `add_state_transition()` 创建 UIState 节点时，增加 `page_type` 和 `semantic_summary` 属性：
   ```cypher
   MERGE (s:UIState {state_id: $state_id})
   ON CREATE SET s.page_type = $page_type, s.semantic_summary = $summary
   ON MATCH SET s.frequency = coalesce(s.frequency, 0) + 1
   ```
2. 新增 `get_pages_by_type(app, page_type)` — 按页面类型查询
3. 新增 `find_similar_pages(page_type, summary, threshold)` — 语义去重查询

**验证**: 
- 创建带 page_type 的 UIState → 查询确认属性存在
- `get_pages_by_type("淘宝", "home")` 返回正确结果

### Task 2.2: 批量导入接口

**目标**: GraphStore 新增 `import_exploration_data()` 方法，批量导入 Phase 1 JSON。

**文件**: `phone_agent/memory/graph_store.py` (+80 行)

**变更**:
1. 新增 `import_exploration_data(pages_file, trajectory_file)`:
   ```
   1. 加载 pages JSON → 为每个 page 创建/合并 UIState 节点
   2. 加载 trajectory JSON → 按 step 顺序创建 Action 节点 + 关系
   3. 创建 ExplorationSession 节点（区分离线探索 vs 在线执行）
   4. 更新 TaskIndex
   ```
2. 使用 Neo4j `apoc` 或批量 `UNWIND` 优化导入性能
3. 新增 `_merge_ui_state(page_info)` — 去重合并逻辑

**验证**:
- 用 Phase 1 产出的 JSON 文件导入 → Neo4j Browser 查看图谱结构
- 重复导入同一批数据 → 确认不创建重复节点

### Task 2.3: 页面语义合并

**目标**: 相同页面类型 + 相似语义的 UIState 节点自动合并，避免图谱膨胀。

**文件**: `phone_agent/memory/graph_store.py` (+60 行)

**变更**:
1. 新增 `_find_or_create_state(page_type, summary, elements)`:
   ```
   1. 按 page_type 过滤候选节点
   2. 计算 summary 的 embedding → 与候选节点做余弦相似度
   3. 相似度 >= 0.90 → 合并（更新 elements, 增加 frequency）
   4. 相似度 < 0.90 → 创建新节点
   ```
2. 新增 `_compute_text_similarity(text1, text2)` — 使用 embedding-3 做文本级相似度
3. 合并策略：`elements` 字段做 key union，`frequency` 累加

**验证**:
- 导入两个相似首页 → 确认合并为一个节点，frequency=2
- 导入首页和搜索结果页 → 确认不合并

### Task 2.4: 动态轨迹查询（代替硬编码 15 步）

**目标**: 将 `get_task_trajectory()` 从硬编码 MATCH 改为可变长度路径查询。

**文件**: `phone_agent/memory/graph_store.py` (重写 ~60 行)

**变更**:
1. 使用 Neo4j 可变长度路径查询：
   ```cypher
   MATCH (t:TaskTarget {target_id: $task_id})-[:STARTS_AT]->(start:UIState)
   MATCH path = (start)-[:NEXT_ACTION]->(a:Action)-[:PRODUCES]->(s:UIState)
   WHERE (s)-[:NEXT_ACTION]->() OR EXISTS { (t)-[:ENDS_AT]->(s) }
   RETURN path LIMIT 1
   ```
   或使用 `apoc.path.expand` 配置 max_level
2. 返回结构保持不变，但去掉 15 步上限
3. 新增 page_type 和 summary 在各 step 中

**验证**:
- 导入 20 步探索轨迹 → 查询正确返回全部 20 步
- 空轨迹 → 返回空 dict（不报错）

### Task 2.5: ExplorationSession 节点类型

**目标**: 区分离线探索轨迹和在线执行轨迹，便于后续分析。

**文件**: `phone_agent/memory/graph_store.py` (+30 行)

**变更**:
1. 新增 `ExplorationSession` 标签的节点：
   ```cypher
   CREATE (es:ExplorationSession:TaskTarget {
     target_id: $id,
     session_type: 'offline_exploration',  // vs 'online_execution'
     app: $app,
     explored_at: $timestamp,
     pages_discovered: $count
   })
   ```
2. 在 `import_exploration_data()` 中创建
3. 在 `find_similar_tasks()` 中增加 `session_type` 过滤选项

**验证**:
- 导入后查询 `MATCH (es:ExplorationSession) RETURN es` 确认节点存在

### Task 2.6: 管线脚本 + CLI

**目标**: 提供一键式管线：探索 → 导入 → 查询验证。

**文件**: 
- `phone_agent/memory/graph_store.py` — `import_exploration_data()`
- `phone_agent/memory/run_explorer.py` — 新增 `--import-to-graph` 参数

**变更**:
1. 在 `run_explorer.py` 中添加 `--import-to-graph` 选项
2. 探索完成后自动调用 `graph_store.import_exploration_data()`
3. 输出导入统计：创建了多少节点、多少关系、合并了多少页面
4. 添加 `--dry-run` 选项（只探索，不导入）

**验证**:
```bash
python -m phone_agent.memory.run_explorer --app 淘宝 --max-steps 10 --import-to-graph
# 期望输出:
#   Imported: 8 UIState nodes, 7 Action nodes, 7 relationships
#   Merged: 2 pages (home, search_result)
#   Created ExplorationSession: 淘宝_20260503_...
```

## 风险点

| 风险 | 缓解 |
|------|------|
| Neo4j 无 apoc 插件，可变长路径查询受限 | 回退方案：将 max_path_length 设为参数，生成动态 MATCH 子句 |
| embedding-3 API 速率限制 | 对 text similarity 做本地缓存，批量调用 |
| 页面去重阈值难以确定 | 先用 0.90，通过测试数据调节 |
| 探索数据量小（1-2 条轨迹），图谱稀疏 | 合并多个 App 的探索数据，或多次运行探索积累 |
